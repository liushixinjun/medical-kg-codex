# Schema 冗余与优化评估报告

评估时间：2026-07-10 23:58:00  
评估对象：`专科知识图谱Schema标准.md` V1.11  
评估范围：主 Schema、解析 SKILL、审计脚本、服务器 Neo4j 实际数据  
本次动作：只读评估，不修改 Schema，不写 Neo4j，不迁移数据

## 一、结论

当前 Schema 不适合“推倒重来”，但需要做一次 V1.12 级别的结构治理。

主要问题不是 Schema 单纯太多，而是三类不一致：

1. **实际库已有数据，但 Schema 未正式收编**  
   教材骨架已经产生 `DefinitionComponent`、`DiagnosisCriteriaComponent`、`Prevention`、`Definition`、`TextbookSection`、`DiseaseClassification` 等实体，但 V1.11 主 Schema 未列入标准实体，导致“文档标准、脚本、服务器数据”不完全一致。

2. **Schema 有预留实体/关系，但当前没有实际数据**  
   如 `ScoringScale`、`PatientState`、`ClinicalEvent`、`Indication`、`TreatmentTiming`、`TimeWindow`、`DrugInteraction`、`AdverseEffect` 等，目前服务器为 0。它们不一定冗余，但应标注为“预留/高级 CDSS 安全层”，不能作为普通批次硬性 required。

3. **历史关系存在大小写双轨**  
   服务器同时存在 `has_symptom` 和 `HAS_SYMPTOM`、`has_treatment_plan` 和 `HAS_TREATMENT_PLAN` 等关系。Schema 应明确小写为唯一标准，历史大写关系只作为待迁移兼容层。

## 三、实际有数据但未被 Schema 正式收编的实体

| 实体 | 实际数量 | 判断 | 建议 |
|---|---:|---|---|
| `DefinitionComponent` | 248 | 有价值 | 收编。用于疾病定义拆解，不应只存在长文本定义 |
| `DiagnosisCriteriaComponent` | 177 | 有价值 | 收编。用于诊断标准明细展示和推理 |
| `Prevention` | 144 | 有价值 | 收编。用于预防措施、二级预防、生活方式管理 |
| `Definition` | 124 | 有条件保留 | 收编但限定用途：来源级定义容器 |
| `TextbookSection` | 73 | 有价值但命名偏窄 | 建议改为通用 `SourceSection`，`TextbookSection` 作为历史兼容 |
| `DiseaseClassification` | 68 | 有价值 | 建议替代 `ClassificationStage` 作为更清晰的标准名 |
| `ClinicalManifestation` | 2 | 冗余 | 不建议收编；迁移到 `Symptom`/`Sign` 或 `SourceSection` |

### 关键判断

`Definition`、`DefinitionComponent` 不是简单冗余。建议分工如下：

```text
Disease.description：
  疾病标准定义，一句话或短段，供前端和 CDSS 快速展示。

Definition：
  某来源、某章节下的定义原文容器，保留教材/指南定义差异。

DefinitionComponent：
  把定义拆成可复用的结构片段，如疾病性质、核心机制、诊断前提、流行病学等。
```

这样既能满足医生看“简明定义”，也能满足你前面要求的“教材第一句话必须在图谱里可追溯、可拆解”。

## 四、Schema 有但当前无数据的实体

| 实体 | 当前数量 | 是否删除 | 建议 |
|---|---:|---|---|
| `ScoringScale` | 0 | 不删 | 风险评分、量表后续需要，如 GRACE、CHA2DS2-VASc |
| `PatientState` | 0 | 暂不删 | 可用于特殊人群，但优先由 `ClinicalRule` 承载条件 |
| `ClinicalEvent` | 0 | 暂不删 | 可用于再入院、复发、猝死等事件，但不要作为普通 required |
| `ClassificationStage` | 0 | 建议调整 | 与实际 `DiseaseClassification` 冲突，名称也容易和 `PathwayStage` 混淆 |
| `Indication` | 0 | 暂不删 | 正式 CDSS 药物/操作安全层需要 |
| `TreatmentTiming` | 0 | 暂不删 | AMI 再灌注、溶栓时间窗等需要 |
| `TimeWindow` | 0 | 暂不删 | 同上，应在指南决策层补抽 |

结论：这些不是全部冗余，更多是“高级 CDSS 安全层尚未建设”。V1.12 应给它们标注成熟度，不允许普通批次因为这些实体没数据就失败。

## 六、字段层问题

V1.11 要求教材骨架知识有 `skeleton_slot` 和 `knowledge_layer`，但服务器实际情况如下：

```text
skeleton_slot：未作为通用属性入库
definition_skeleton_slot：68 个 Disease 节点有
knowledge_layer：3129 个节点有
knowledge_layer 取值：
  textbook_skeleton：1860
  evidence：1269
```

问题：

1. `skeleton_slot` 标准要求与实际入库字段不一致。
2. `knowledge_layer` 实际值 `textbook_skeleton/evidence` 与 V1.11 枚举 `textbook_core/guideline_supplement/guideline_decision/...` 不一致。
3. 教材骨架节点 `schema_version/review_status/display_name/entityCategory` 有 3129 个空值，说明骨架导入层仍有字段补齐问题。

建议：

```text
V1.12 必须统一字段：
1. 教材骨架实体统一写 skeleton_slot。
2. Disease 定义补充字段可保留 definition_skeleton_slot，但不得替代通用 skeleton_slot。
3. knowledge_layer 枚举要二选一：
   A. 迁移实际值到标准值：textbook_skeleton -> textbook_core；
   B. 修改 Schema 枚举收编 textbook_skeleton。
推荐 A：迁移到标准值，避免标准继续膨胀。
```

## 七、建议的 V1.12 优化方案

### 方案 A：只加兼容说明，不改结构

优点：风险最低。  
缺点：Schema 与实际数据继续不一致，后续批次仍会反复踩坑。

不推荐。

### 方案 B：结构治理，不推倒重来

这是推荐方案。

核心动作：

1. 正式收编有价值的实际实体：
   - `Definition`
   - `DefinitionComponent`
   - `DiagnosisCriteriaComponent`
   - `Prevention`
   - `SourceSection`

2. 调整/兼容命名：
   - `DiseaseClassification` 作为新的标准名；
   - `ClassificationStage` 标记为 legacy alias，不再新建；
   - `TextbookSection` 迁移为 `SourceSection`；
   - `ClinicalManifestation` 不收编，迁移到 `Symptom/Sign` 或 `SourceSection`。

3. 明确实体成熟度：
   - `core_required`：Disease、Symptom、Sign、Exam、LabTest、DiagnosisCriteria、TreatmentPlan、Evidence 等；
   - `cdss_required_when_applicable`：ClinicalPathway、PathwayStage、ClinicalRule、RecommendationStatement、TimeWindow、Contraindication；
   - `drug_safety_advanced`：DrugInteraction、AdverseEffect；
   - `reserved_optional`：ClinicalEvent、PatientState、ScoringScale 等。

4. 关系统一小写：
   - 新批次只允许小写标准关系；
   - 大写历史关系进入迁移清单；
   - 前端短期兼容，后端长期只按小写查。

5. 字段统一：
   - 补齐 `skeleton_slot`；
   - 统一 `knowledge_layer` 枚举；
   - 骨架节点补齐 `schema_version/review_status/display_name/entityCategory`。

### 方案 C：重建 Schema 分层本体

把 Schema 拆成：

```text
临床事实本体
来源证据本体
CDSS 决策本体
流程引擎接口本体
```

优点：长期最干净。  
缺点：当前已入库 43020 节点、140183 关系，重建成本高，会影响 Trae 前端和已解析批次。

当前不建议。

## 九、是否可以优化

可以优化，而且应该优化。

但优化方向不是“删掉没数据的实体”，而是：

```text
实际有用但未收编的：收编；
实际重复或命名冲突的：合并/废弃；
当前没数据但 CDSS 高级能力需要的：标记为预留，不作为普通批次硬门槛；
历史大小写关系：迁移到小写标准；
字段实际值与标准不一致：统一枚举和入库字段。
```

这能解决当前最大问题：Schema、脚本、服务器、前端查询口径不完全一致。
