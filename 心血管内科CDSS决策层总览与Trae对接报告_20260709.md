# 心血管内科 CDSS 决策层总览与 Trae 对接报告

生成时间：2026-07-09 22:20:00

用途：给 Trae/前端/后端说明“图谱怎么用于专科 CDSS”，避免只拿 Schema 不知道怎么查、怎么展示、怎么和患者 EMR 结合。

## 2. CDSS 正确使用链路

图谱不是“一次性把疾病下所有节点圈出来”。专科 CDSS 应按下面链路取数：

```text
Disease 疾病
  -> has_clinical_pathway -> ClinicalPathway 专病诊疗路径
  -> has_pathway_stage -> PathwayStage 诊疗阶段
  -> has_stage_rule -> ClinicalRule 阶段规则
  -> has_recommendation_statement -> RecommendationStatement 推荐陈述
  -> recommends_action / blocks_action -> 具体动作：检查、检验、药物、手术、治疗方案、随访等
  -> supported_by_evidence -> Evidence 单条证据
  -> based_on_guideline -> Guideline 指南/教材/共识来源
```

医生界面应展示“当前阶段、为什么触发、推荐什么、依据哪条证据”。不要展示疾病级一箩筐证据。

## 3. 前端页面建议

### 3.1 疾病总览页

左侧疾病树展示：疾病大类 -> 疾病。每个疾病展示：完整率、路径数、推荐数、证据数。点击疾病后进入专病流程，不再一次性展示所有检查/用药/手术。

### 3.2 专病流程页

按阶段卡片展示：

1. 诊断确认：诊断标准、必要检查/检验、鉴别诊断。
2. 风险分层：评分、危险因素、并发症、高危标志。
3. 治疗决策：药物、手术、介入、综合治疗、禁忌/排除。
4. 随访管理：复查、随访频率、预后、患者教育。

每个阶段下展示规则卡片，每条规则卡片下展示推荐陈述。推荐陈述才是 CDSS 推荐的最小展示单元。

### 3.3 推荐卡片字段

| 区域 | 字段 | 说明 |
|---|---|---|
| 推荐内容 | `RecommendationStatement.statement_text` | 医生首先看到的一句话推荐 |
| 推荐动作 | `recommends_action` / `blocks_action` 下游节点 | 具体检查、检验、用药、手术、治疗方案、随访 |
| 推荐强度 | `recommendation_class` | 例如 I、IIa、未分级/指南未显式标注 |
| 证据等级 | `evidence_level` | 例如 A、B、C、未分级 |
| 主证据来源 | `primary_source_name`、`primary_source_page` | 指南/教材名称和页码 |
| 证据摘要 | `primary_evidence_summary` | 给医生看的简短依据 |
| 原文摘录 | `primary_evidence_raw_excerpt` | 展开后查看，不能默认刷屏 |
| 审核状态 | `clinical_review_status`、`formal_cdss_ready` | 测试推荐和正式上线分层 |

## 4. 后端接口建议

### 4.1 获取可用疾病

接口：`GET /api/kg/cdss/diseases?subject=心血管内科`

返回字段：`disease_code`、`disease_name`、`category`、`pathway_count`、`recommendation_count`、`evidence_count`、`guideline_count`。

```cypher
MATCH (d:Disease:KGNode)-[:has_clinical_pathway]->(p:ClinicalPathway:KGNode)
WHERE p.batch_id STARTS WITH 'BATCH-CARD-' AND p.batch_id CONTAINS '-CDSS-20260709-001'
OPTIONAL MATCH (p)-[:has_pathway_stage]->(:PathwayStage)-[:has_stage_rule]->(:ClinicalRule)-[:has_recommendation_statement]->(rec:RecommendationStatement)
OPTIONAL MATCH (rec)-[:supported_by_evidence]->(e:Evidence)
OPTIONAL MATCH (rec)-[:based_on_guideline]->(g:Guideline)
RETURN d.code AS disease_code, d.name AS disease_name, coalesce(d.disease_category,d.category) AS category,
       count(DISTINCT p) AS pathway_count, count(DISTINCT rec) AS recommendation_count,
       count(DISTINCT e) AS evidence_count, count(DISTINCT g) AS guideline_count
ORDER BY category, disease_name
```

### 4.2 获取某疾病完整诊疗路径

接口：`GET /api/kg/cdss/pathway?diseaseCode=DIS-CARD-CAD-AMI`

```cypher
MATCH (d:Disease:KGNode {code:$diseaseCode})-[:has_clinical_pathway]->(p:ClinicalPathway:KGNode)
MATCH (p)-[:has_pathway_stage]->(s:PathwayStage:KGNode)
OPTIONAL MATCH (s)-[:has_stage_rule]->(r:ClinicalRule:KGNode)
OPTIONAL MATCH (r)-[:has_recommendation_statement]->(rec:RecommendationStatement:KGNode)
OPTIONAL MATCH (rec)-[:recommends_action]->(a:KGNode)
OPTIONAL MATCH (rec)-[:blocks_action]->(ba:KGNode)
OPTIONAL MATCH (rec)-[:supported_by_evidence]->(e:Evidence:KGNode)
OPTIONAL MATCH (rec)-[:based_on_guideline]->(g:Guideline:KGNode)
RETURN d.code AS disease_code, d.name AS disease_name,
       p.code AS pathway_code, p.name AS pathway_name,
       s.code AS stage_code, s.name AS stage_name, s.stage_order AS stage_order, s.stage_type AS stage_type,
       r.code AS rule_code, r.name AS rule_name,
       rec.code AS recommendation_code, rec.statement_text AS statement_text,
       rec.recommendation_class AS recommendation_class, rec.evidence_level AS evidence_level,
       rec.primary_source_name AS primary_source_name, rec.primary_source_page AS primary_source_page,
       rec.primary_evidence_summary AS primary_evidence_summary,
       collect(DISTINCT {code:a.code,name:a.name,type:a.entityType}) AS recommended_actions,
       collect(DISTINCT {code:ba.code,name:ba.name,type:ba.entityType}) AS blocked_actions,
       collect(DISTINCT {code:e.code,summary:e.summary,page:e.page}) AS evidences,
       collect(DISTINCT {code:g.code,name:g.name}) AS guidelines
ORDER BY stage_order, rule_code, recommendation_code
```

### 4.3 获取单条推荐的证据详情

接口：`GET /api/kg/cdss/recommendation/{recommendationCode}/evidence`

```cypher
MATCH (rec:RecommendationStatement:KGNode {code:$recommendationCode})
OPTIONAL MATCH (rec)-[:recommends_action]->(a:KGNode)
OPTIONAL MATCH (rec)-[:blocks_action]->(ba:KGNode)
OPTIONAL MATCH (rec)-[:supported_by_evidence]->(e:Evidence:KGNode)
OPTIONAL MATCH (rec)-[:based_on_guideline]->(g:Guideline:KGNode)
RETURN rec.code AS recommendation_code, rec.statement_text AS statement_text,
       rec.recommendation_class AS recommendation_class, rec.evidence_level AS evidence_level,
       rec.primary_source_name AS primary_source_name, rec.primary_source_page AS primary_source_page,
       rec.primary_evidence_summary AS primary_evidence_summary,
       rec.primary_evidence_raw_excerpt AS primary_evidence_raw_excerpt,
       collect(DISTINCT {code:a.code,name:a.name,type:a.entityType}) AS recommended_actions,
       collect(DISTINCT {code:ba.code,name:ba.name,type:ba.entityType}) AS blocked_actions,
       collect(DISTINCT {code:e.code,summary:e.summary,page:e.page,raw_excerpt:e.raw_excerpt}) AS evidences,
       collect(DISTINCT {code:g.code,name:g.name}) AS guidelines
```

## 5. 与患者 EMR / 规则引擎的组合

图谱负责“医学知识和证据链”，规则引擎负责“结合患者当前数据判断是否触发”。推荐落地链路：

```text
EMR 患者数据 -> 标准化患者事实 patient_facts -> 匹配 ClinicalRule -> 返回 RecommendationStatement -> 展示动作和证据
```

患者事实建议统一成：

```json
{
  "patient_id": "住院号/就诊号",
  "disease_code": "DIS-CARD-CAD-AMI",
  "facts": [
    {"field_code":"ECG_ST_ELEVATION","field_name":"ST段抬高","value":true,"source":"心电图"},
    {"field_code":"TROPONIN_POSITIVE","field_name":"肌钙蛋白升高","value":true,"source":"检验"},
    {"field_code":"BLEEDING_RISK_HIGH","field_name":"出血风险高","value":false,"source":"评估"}
  ]
}
```

当前图谱已经支持“推荐链 + 证据链”。患者动态触发还需要后端把患者字段映射到规则条件；不能让前端自己从疾病节点下所有内容里猜。

## 6. 前端必须避免的错误

1. 不要从 `Disease -> Evidence` 或疾病证据池展示治疗依据。
2. 不要从 `Action -> Evidence` 反推当前推荐依据。动作是可复用实体，依据必须落在 `RecommendationStatement`。
3. 不要把 `TreatmentPlan` 当成流程阶段；阶段是 `PathwayStage`。
4. 不要把“指南与证据”作为一个大列表默认展示给医生；只在推荐卡片中展示当前推荐的主证据，更多证据点击展开。
5. 不要只按 `batch_id` 判断疾病大类；跨谱系疾病如缺血性心肌病要按 `Disease.code` 和 `disease_category` 展示。

## 7. 推荐卡片样例

### 心房颤动｜心房颤动诊断确认阶段

- 推荐：疑似心房颤动时，应结合心房颤动诊断标准、12导联心电图、事件记录仪、动态心电图、心脏CT等进行诊断确认和必要鉴别。
- 动作：凝血功能检查, 动态心电图, 事件记录仪
- 推荐等级：未分级/指南未显式标注
- 证据等级：未分级/指南未显式标注
- 主证据：心房颤动诊断和治疗中国指南 2023.pdf，第 3 页
- 摘要：疑似心房颤动时，应结合心房颤动诊断标准、12导联心电图、事件记录仪、动态心电图、心脏CT等进行诊断确认和必要鉴别。

### 心房颤动｜心房颤动风险分层阶段

- 推荐：心房颤动确诊或高度疑似后，应结合CHA2DS2-VASc评分、EHRA症状分级、心房颤动风险分层、冠心病、吸烟等进行风险分层和高危因素识别。
- 动作：心力衰竭, 心力衰竭加重, 慢性肾脏病
- 推荐等级：未分级/指南未显式标注
- 证据等级：未分级/指南未显式标注
- 主证据：中国心房颤动管理指南（2025）.pdf，第 10 页
- 摘要：心房颤动确诊或高度疑似后，应结合CHA2DS2-VASc评分、EHRA症状分级、心房颤动风险分层、冠心病、吸烟等进行风险分层和高危因素识别。

### 心房颤动｜心房颤动治疗决策阶段

- 推荐：心房颤动治疗应根据病情严重程度、禁忌证和患者特征选择复律治疗、导管消融、左心耳封堵、β受体阻滞剂、低分子肝素等。
- 动作：导管消融, 利伐沙班, 左心耳封堵
- 推荐等级：未分级/指南未显式标注
- 证据等级：未分级/指南未显式标注
- 主证据：中国心房颤动管理指南（2025）.pdf，第 17 页
- 摘要：心房颤动治疗应根据病情严重程度、禁忌证和患者特征选择复律治疗、导管消融、左心耳封堵、β受体阻滞剂、低分子肝素等。

### 心房颤动｜心房颤动随访管理阶段

- 推荐：心房颤动稳定期或治疗后，应按心房颤动随访方案、心房颤动预后进行随访和复评。
- 动作：心房颤动随访方案, 心房颤动预后
- 推荐等级：Ⅱa
- 证据等级：C
- 主证据：中国心房颤动管理指南（2025）.pdf，第 7 页
- 摘要：心房颤动稳定期或治疗后，应按心房颤动随访方案、心房颤动预后进行随访和复评。

## 8. 给 Trae 的开发任务

1. 数据总览页增加“CDSS 决策层”统计，不只展示实体/关系总数。
2. 图谱探索页点击疾病后，默认进入专病流程视图：阶段 -> 规则 -> 推荐陈述 -> 动作/证据。
3. 推荐卡片必须以 `RecommendationStatement` 为根节点查询证据。
4. 证据区默认只显示当前推荐主证据：指南名称、页码、推荐等级、证据等级、原文摘要；完整证据点击展开。
5. 后端接口按本报告第 4 节提供，不要让前端拼复杂 Cypher。

## 9. 当前限制

当前数据为测试推荐层，`formal_cdss_ready=false`。可以用于专科 CDSS 原型、测试库展示、临床使用效果审核；不得直接作为正式自动推荐上线。
