## 1. 总原则

前端必须区分两类内容：一类是“知识浏览”，用于看疾病有哪些症状、检查、治疗方案、指南证据；另一类是“正式 CDSS 推荐”，用于医生诊疗时给出当前患者应该做什么、不能做什么、依据哪份指南。正式推荐只能来自已经入库的推荐来源裁决和推荐陈述，不能从疾病下面的治疗方案、药物、检查、手术节点直接推断。

## 2. 正式 CDSS 推荐区唯一读取链路

正式推荐区只走这一条链路：

```text
疾病 Disease
  -> 推荐来源裁决 SourceAdjudication
  -> 推荐陈述 RecommendationStatement
  -> 推荐动作 Action
  -> 主证据 Evidence
  -> 主指南 Guideline
```

中文理解：

| 图谱对象 | 中文含义 | 前端用途 |
|---|---|---|
| Disease | 疾病 | 当前专病，例如急性心肌梗死 |
| SourceAdjudication | 推荐来源裁决 | 多份指南里最终听哪份、为什么听这份 |
| RecommendationStatement | 推荐陈述 | 医生端看到的一条正式推荐 |
| Action | 推荐动作 | 具体要做或不能做的检查、治疗、手术、用药 |
| Evidence | 主证据 | 支持这条推荐的原文证据 |
| Guideline | 主指南 | 这条推荐主要依据的指南或共识 |

## 3. AMI 正式推荐查询示例

示例疾病：急性心肌梗死  
疾病编码：`DIS-CARD-CAD-AMI`

```cypher
MATCH (d:KGNode {code:$disease_code})
MATCH (d)-[:has_source_adjudication]->(adj:KGNode {entityType:'SourceAdjudication'})
MATCH (adj)-[:decides_recommendation]->(rec:KGNode {entityType:'RecommendationStatement'})
MATCH (adj)-[ar:recommends_action|blocks_action]->(action:KGNode)
MATCH (adj)-[:derived_from]->(ev:KGNode {entityType:'Evidence'})
MATCH (adj)-[:uses_primary_guideline]->(gl:KGNode {entityType:'Guideline'})
WHERE coalesce(adj.cdss_use_status,'') = '正式推荐'
  AND coalesce(rec.cdss_use_status,'') = '正式推荐'
  AND coalesce(adj.primary_evidence_code,'') <> ''
  AND coalesce(adj.recommendation_class,'') <> ''
  AND coalesce(adj.evidence_level,'') <> ''
  AND coalesce(adj.action_code,'') <> ''
RETURN
  adj.code AS source_adjudication_code,
  adj.clinical_question AS clinical_question,
  adj.clinical_scenario AS clinical_scenario,
  rec.code AS recommendation_code,
  rec.recommendation_text AS recommendation_text,
  action.code AS action_code,
  coalesce(action.display_name, action.preferred_name, action.name, action.code) AS action_name,
  action.entityType AS action_type,
  type(ar) AS action_relation,
  adj.primary_guideline_code AS primary_guideline_code,
  adj.primary_guideline_name AS primary_guideline_name,
  adj.primary_evidence_code AS primary_evidence_code,
  adj.recommendation_class AS recommendation_class,
  adj.evidence_level AS evidence_level,
  adj.conflict_status AS conflict_status,
  adj.supporting_guidelines AS supporting_guidelines,
  coalesce(ev.source_page, ev.page, ev.page_number) AS evidence_page,
  coalesce(ev.evidence_text, ev.original_text, ev.name) AS evidence_text
ORDER BY adj.code
```

参数：

```json
{
  "disease_code": "DIS-CARD-CAD-AMI"
}
```

## 4. 前端推荐卡片展示格式

医生端每条推荐卡片建议固定展示这些字段：

| 字段 | 中文显示 | 说明 |
|---|---|---|
| `clinical_question` | 临床问题 | 例如“STEMI再灌注策略” |
| `recommendation_text` | 推荐内容 | 医生真正要看的推荐结论 |
| `action_name` | 推荐动作 | 例如“经皮冠状动脉介入治疗” |
| `action_relation` | 动作性质 | `recommends_action` 表示建议执行；`blocks_action` 表示阻断/禁忌 |
| `primary_guideline_name` | 主依据指南 | 当前推荐主要依据哪份指南 |
| `recommendation_class` | 推荐等级 | 如果原文无 I/A 分级，则显示“专家共识推荐、定义标准、禁忌阻断规则”等 |
| `evidence_level` | 证据等级 | 如果原文无 A/B/C 分级，则显示“专家共识证据、国际通用定义标准”等 |
| `evidence_page` | 页码 | 原文页码 |
| `evidence_text` | 原文摘要 | 只展示支持当前推荐的主证据摘要 |
| `conflict_status` | 冲突状态 | 无冲突才可默认进入正式推荐；待裁决只能提示不能主动推荐 |

## 7. 前端禁止用法

下面这些查询只能用于图谱探索或知识浏览，不能作为正式 CDSS 推荐：

```cypher
MATCH (d)-[:has_treatment_plan]->(plan)
RETURN plan
```

```cypher
MATCH (d)--(anything)
RETURN anything
```

```cypher
MATCH (d)--(e:KGNode {entityType:'Evidence'})
RETURN e
```

原因：

- 疾病直连治疗方案只说明“这个病可能涉及这个治疗”，不说明当前患者该不该做。
- 疾病邻居节点会把症状、检查、并发症、指南、证据全部混在一起。
- 疾病证据池是一堆资料来源，不能代表某一条推荐的主依据。

## 8. 正式推荐区硬闸门

前端正式推荐区必须过滤：

```text
cdss_use_status 必须等于 正式推荐
primary_evidence_code 不能为空
recommendation_class 不能为空
evidence_level 不能为空
action_code 不能为空
conflict_status 不能是 待裁决
```

如果不满足以上条件，只能进入“知识浏览 / 待审核 / 数据问题提示”，不能进入医生主动推荐区。

## 9. 阻断类推荐的展示

`blocks_action` 不是普通推荐，它表示“有禁忌或风险时不要做”。前端必须用不同视觉样式展示，例如：

```text
风险阻断：存在溶栓禁忌证时，不推荐溶栓治疗。
依据：2018 STEMI院前溶栓治疗中国专家共识.pdf
证据：EVD-B9B303B0E1B921BC095F-AMI
```

阻断类推荐不能和普通建议混在同一个“推荐治疗”列表里，否则医生容易误解。

注意：前端不要自己用关键词判断阻断。比如“根据适应证、禁忌证选择治疗”不是阻断，它仍可能是建议执行；只有图谱关系是 `blocks_action` 时才显示为禁忌/阻断。

## 10. 图谱探索页与医生端的边界

图谱探索页可以展示更宽的疾病知识，包括症状、体征、检查、治疗方案、药物、指南、证据、并发症等，但必须标明“知识浏览”。医生端正式 CDSS 推荐区只展示经过来源裁决的推荐陈述。两个区域可以共用图谱数据，但查询入口和展示口径必须分开。
