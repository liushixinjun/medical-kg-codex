# Evidence 证据层瘦身与入库性能优化报告

更新时间：2026-07-15 23:10:00
范围：服务器 Neo4j 当前全库 Evidence 证据节点与证据关系
结论：Evidence 节点需要保留，但必须去重、分层展示、限制长文本入库负载。当前不建议直接删除服务器节点，应先做“去重预演 → 关系迁移预演 → 小批量正式迁移”。

## 1. 当前问题

| 项目 | 当前数量 | 人话解释 | 影响 |
|---|---:|---|---|
| Evidence 总数 | 44,440 | 图谱里有 4.4 万个证据节点 | 数量明显偏大 |
| 空证据文本 | 1,269 | 证据节点没有原文内容 | 医生端不可用，需要清理或补证据 |
| 超过 5,000 字证据 | 2,874 | 单个证据节点文本过长 | 入库慢，查询慢，前端展示困难 |
| 超过 10,000 字证据 | 1,240 | 明显不适合直接作为医生端展示文本 | 需要摘要化、定位化 |
| 孤立证据 | 329 | 证据没有被任何临床实体引用 | 大概率是历史残留或无效证据 |
| 重复证据分组 | 7,081 | 相同来源、页码、原文被建成多组重复节点 | 是证据膨胀主因 |
| 重复证据节点 | 33,703 | 重复组内包含的证据节点总数 | 需要治理 |
| 理论可减少节点 | 26,622 | 如果每组只保留 1 个主证据节点，可减少的节点数 | 约可减少 60% Evidence 节点 |

## 2. 根因判断

主要问题不是 Schema 实体设计错误，而是生成策略不合理：

1. 同一份指南、同一页、同一段原文，被按不同疾病重复创建 Evidence 节点。
2. 证据节点 code 里带了疾病后缀，导致本来应该共享的证据被拆成多个节点。
3. 大段原文直接写入 Neo4j，批量入库时 HTTP 请求体过大，导致 500 错误或超时。
4. 医生端如果按“疾病 → 全部证据”展示，会看到一箩筐证据，无法判断某条推荐到底依据哪条指南。

## 3. Evidence 证据层保留原则

Evidence 不是冗余实体，不能简单删除。它承担三个作用：

1. 溯源：说明某条诊断、治疗、检查、随访建议来自哪份指南、哪一页、哪一段。
2. 审核：给临床专家、药师、前端同事核对推荐依据。
3. 责任边界：区分“教材基础知识”“指南推荐”“专家共识”“外部权威补源”。

但 Evidence 不应该按疾病重复建，也不应该把长篇全文都直接堆给医生看。

## 4. 瘦身目标结构

### 4.1 主证据节点

同一来源、同一页、同一段原文，只保留一个 Evidence 主节点。

建议生成唯一键：

```text
证据唯一键 = 来源文件名 + 页码 + 原文规范化文本 的哈希值
```

说明：

- “哈希值”就是把一段长文本计算成一个固定短编号，用来判断是否重复。
- 不再按疾病后缀生成多个 Evidence。
- 疾病、推荐、规则的差异放在关系上，不放在 Evidence 节点重复上。

### 4.2 推荐与证据关系

CDSS 推荐、诊断规则、治疗方案、检查建议等临床实体，通过 `supported_by_evidence` 指向 Evidence。

医生端展示时，不展示疾病下所有 Evidence，只展示当前推荐直接关联的 1-3 条主证据：

| 医生看到的推荐 | 展示证据 |
|---|---|
| 急诊 PCI | 急诊 PCI 这条推荐直接关联的指南名称、页码、推荐等级、证据等级、原文摘要 |
| 溶栓治疗 | 溶栓治疗这条推荐直接关联的适应证、禁忌证、药物用法证据 |
| 肌钙蛋白复查 | 肌钙蛋白复查规则直接关联的时间窗和阈值证据 |

## 5. 长文本治理规则

| 字段 | 建议存放位置 | 说明 |
|---|---|---|
| 证据摘要 | Neo4j Evidence 节点 | 医生端和审核端优先展示 |
| 指南名称 | Neo4j Evidence 节点 | 必须保留 |
| 页码/段落/章节 | Neo4j Evidence 节点 | 必须保留 |
| 推荐等级 | RecommendationStatement 或 Evidence | 有则保留 |
| 证据等级 | RecommendationStatement 或 Evidence | 有则保留 |
| 原文短摘录 | Neo4j Evidence 节点 | 控制长度，便于核对 |
| 超长原文 | 本地证据文件或文档库 | 不建议整段塞进图谱查询主路径 |
| 原文哈希 | Neo4j Evidence 节点 | 用于去重和审计 |

当前服务器已有长文本，不能直接截断。下一步应先生成迁移映射和备份文件，再执行正式瘦身。

## 6. 入库性能优化规则

| 场景 | 推荐批量大小 | 说明 |
|---|---:|---|
| 普通节点 | 200-500 | 一般实体体积小，可以较大批量 |
| Evidence 节点 | 20-50 | 原文长，必须小批量 |
| 普通关系 | 200-500 | 关系属性少，可以较大批量 |
| 证据关系 | 50-100 | 数量大，建议中小批量 |
| 失败处理 | 自动二分拆批 | 失败时自动拆成更小批次，定位坏数据 |
| 日志 | 每批写进度日志 | 防止中途断线后不知道进度 |

本轮 5 批次入库已经验证：Evidence 小批量写入可以绕开长文本导致的 HTTP 500 和超时问题。

## 7. 推荐执行顺序

### 第一步：生成端改造

新批次生成 Evidence 时，先计算“证据唯一键”。如果本批次或服务器已有相同证据，则复用主证据，不再创建重复 Evidence 节点。

### 第二步：服务器去重预演

只读扫描服务器，输出：

- 重复组清单；
- 每组建议保留的主证据节点；
- 需要迁移的旧证据节点；
- 受影响关系数量；
- 可能风险。

该阶段不写库。

### 第三步：正式迁移

确认预演结果后，再执行：

1. 将重复 Evidence 的入边迁移到主证据节点；
2. 将重复 Evidence 的出边迁移到主证据节点；
3. 保留主证据节点的来源、页码、摘要、原文哈希；
4. 删除或归档重复 Evidence；
5. 执行服务器 postcheck。

## 8. 当前是否可以直接瘦身入库

不建议现在直接删除或合并 26,622 个 Evidence 节点。

原因：

1. Evidence 被 129,262 条 `supported_by_evidence` 关系引用，直接动库风险高。
2. 有 23,869 条 `guideline_has_evidence` 关系，需要同步迁移。
3. 需要先确认重复组主证据选择规则，避免把不同推荐等级、不同上下文的证据误合并。

当前可以立即做的是：

- 固化新批次 Evidence 生成规则；
- 生成服务器去重预演清单；
- 优化入库脚本批量策略；
- 前端只按“推荐/规则 → 主证据”展示，不按“疾病 → 全部证据”展示。

## 9. 对 Trae/前端的展示要求

医生端不要显示“某疾病全部 Evidence 列表”。正确显示方式：

```text
疾病 → 诊疗阶段 → 当前推荐/规则 → 直接支持该推荐的主证据
```

每条推荐最多默认展示 1-3 条证据：

- 指南名称；
- 页码/章节；
- 推荐等级；
- 证据等级；
- 原文摘要；
- 点击后再展开完整来源。

## 10. 本轮结论

Evidence 层可以优化，且优化空间很大；但不能用“直接删节点”的方式处理。正确做法是先从生成端防止继续膨胀，再对历史 Evidence 做可回滚的去重迁移。
