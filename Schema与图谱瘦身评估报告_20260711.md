## 3. 当前图谱实体是否臃肿

从服务器实际数据看，实体类型总体没有严重臃肿，但有两类需要观察：

### 3.1 低频实体

| entityType | 数量 | 判断 |
|---|---:|---|
| `Specialty` | 1 | 顶层学科根节点，低频正常，不建议删 |
| `Contraindication` | 9 | 暂时低频，但 CDSS 禁忌/排除条件需要，保留观察 |

结论：低频不等于冗余。`Specialty` 是图谱顶层锚点；`Contraindication` 是 CDSS 关键语义，不能因为数量少就删。

### 3.2 可能重复表达的实体

| 候选 | 判断 | 建议 |
|---|---|---|
| `Definition` 与 `DefinitionComponent` | 非冗余 | `Definition` 是定义标题/总述，`DefinitionComponent` 是可拆解要点 |
| `DiagnosisCriteria` 与 `DiagnosisCriteriaComponent` | 非冗余 | 前者是诊断标准标题，后者是推理明细 |
| `DiseaseClassification` 与历史 `ClassificationStage` | 有兼容负担 | 新批次只用 `DiseaseClassification`；历史 `ClassificationStage` 只读兼容 |
| `SourceSection` 与 `Evidence` | 非冗余 | `SourceSection` 是来源章节锚点，`Evidence` 是证据片段 |

## 4. 当前图谱关系是否臃肿

### 4.1 已清理

`USES_MEDICATION`、`HAS_PROCEDURE`、`HAS_CLINICAL_MANIFESTATION` 已清理为 0。

### 4.2 需要下一轮专项评估

| 关系 | 数量 | 初步判断 |
|---|---:|---|
| `has_recommended_action` | 1466 | 阶段到动作的路径快捷边 |
| `recommends_action` | 2740 | 规则/推荐陈述到动作的推理推荐边 |

这两个名字相近，但语义不同：

```text
PathwayStage -> has_recommended_action -> Action
表示：某诊疗阶段默认可展示哪些动作。

ClinicalRule / RecommendationStatement -> recommends_action -> Action
表示：患者满足规则后，正式推荐哪个动作。
```

结论：不能直接删除 `has_recommended_action`。但命名容易误导，建议下一版 Schema 将其改名为更清楚的：

```text
PathwayStage -> stage_has_available_action -> Action
```

或保留但明确：

```text
has_recommended_action 仅用于路径阶段展示，不作为患者级推荐。
```

| `exam_has_indicator` | 6 | 检查-指标拆解，后续会增加 |

## 5. 推荐的瘦身原则

以后不要按“数量少就删”，按下面四类判断：

| 类型 | 处理 |
|---|---|
| 旧命名、弱语义、前端可能误用 | 归档后删除 |
| 低频但临床语义关键 | 保留，写清用途 |
| 名称相近但层级不同 | 不删，改名或补说明 |
| 只用于历史兼容 | 标记 legacy，禁止新批次生成 |
