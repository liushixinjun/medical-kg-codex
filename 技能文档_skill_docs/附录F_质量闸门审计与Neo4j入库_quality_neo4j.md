# 附录F：质量闸门、审计与Neo4j入库

## 2026-07-15 23:35 新增｜Evidence 复用闸门正式接入 G2

后续所有新批次执行 `scripts/audit_graph_instance.py` 时，必须同步检查 Evidence 证据复用质量：

1. 同一来源文件、同一页码、同一段原文只能生成 1 个共享 Evidence 节点。
2. Evidence 必须带 `evidence_key`，中文解释为“证据唯一键”，由来源文件、页码、原文规范化文本计算得到。
3. Evidence 编号必须使用 `EVD-SHARED-` 前缀；原始旧编号只能保留在 `original_evidence_ids`，不能继续作为主节点编号。
4. 出现重复证据组、缺证据唯一键、非共享编号、共享编号不一致时，G2 质量闸门直接失败。

本规则已接入本地 G2 审计入口，不再只依赖单独脚本人工检查。

## 2026-07-16 00:25 新增｜历史 Evidence 迁移分片规则

历史 Evidence 去重迁移不得长事务全量执行，必须分片：

1. 每片默认 100 个重复 Evidence 节点。
2. 每片必须输出迁移明细和关系备份。
3. 每片完成后，重复节点剩余关系必须为 0。
4. 不处理空证据文本；空证据文本应单独作为来源质量问题治理。
5. 不物理删除节点，只迁关系、标记 `deprecated=true` 和 `duplicate_replaced_by`。

## 2026-07-15 23:10 新增｜Evidence 证据层瘦身与性能规则

后续所有批次必须增加 Evidence 证据层检查：

1. 同一来源文件、同一页码、同一段原文不得重复创建 Evidence 节点。
2. 新批次必须生成“证据唯一键”，用于复用主证据节点。
3. 医生端不得按疾病展示全量 Evidence 池，只能按“推荐/规则 -> 主证据”展示。
4. Evidence 节点入库批量建议 20-50；证据关系入库批量建议 50-100；失败时必须自动拆批并记录进度日志。
5. 服务器历史 Evidence 去重必须先输出预演映射，不得直接批量删除或合并证据节点。

当前专项报告：`项目管理中心_project_management/35_Evidence证据层瘦身与入库性能优化报告_20260715.md`。

## 2026-07-15 14:34 新增｜肺动脉高压入库后复测暴露的固定闸门

后续所有批次必须增加两项固定检查：

1. 诊断标准必须有明细：`DiagnosisCriteria` 不能只有标题，必须至少有 1 条 `has_diagnostic_component`。
2. 临床规则必须做章节主题归属：`ClinicalRule` 必须同时符合疾病范围、原文章节主题、规则主语义，不能只靠关键词命中。

写库后仍以服务器 postcheck 为最终结果：服务器阻断项不为 0 时，不能在台账写“已入库并通过”。

后续所有新病种、旧批次精修、补丁修复，只要写入 Neo4j，入库后必须统一执行：

旧的零散 postcheck 脚本仅作为历史排查资料，不再作为新批次默认入口。

## 2026-07-14 15:27 新增｜主数据质量闸门

后续新病种、旧批次精修和服务器入库后复核，必须执行主数据质量闸门：

```powershell
D:\Program Files Ai\python-venvs\medical-kg\Scripts\python.exe `
  公共执行层_kg_pipeline\主数据质量闸门_master_data_gate.py `
  --connection-file 图谱数据库链接.txt `
  --output-dir 心血管内科文献集合\00_全局质量体检_global_quality_audit\YYYYMMDD_主数据质量闸门 `
  --fail-on-blocking
```

必须为 0 的检查项：

| 检查项 | 必须为 0 的原因 |
|---|---|
| 同病种同类型同名重复直连 | 避免前端重复展示和 CDSS 推理入口不稳定 |
| 诊断标准无明细 | 诊断标准必须能下钻到具体条件 |
| 临床规则章节主题不一致 | 避免相邻疾病章节内容误入本病 CDSS 规则 |
| 孤儿诊断标准 | 零关系节点不得进入正式图谱 |
| 治疗方案无下游 | 治疗方案必须能下钻到药物、操作、子方案、证据或路径动作 |
| 已替换重复节点仍被引用 | 已归并的重复节点不得继续被疾病、路径或规则引用 |

术语说明：标准主节点 = 同一临床概念最终保留和复用的主节点；重复节点的关系必须迁移到标准主节点。

## 3. G3 写库条件

默认不写库。允许写库时必须有：

1. 用户授权。
2. G2 通过。
3. 导入脚本、导入数据、回滚策略。
4. 服务器 postcheck 脚本和输出目录。

历史候选池、Trae 候选、人工临时补丁不得直接进入 G3。必须先回到教材/指南原文，生成当前批次本地 JSONL/CSV，并通过 G2。

## 4. 服务器 postcheck

服务器统计必须从 Neo4j 实际查询，不用本地文件替代。输出要讲人话：

```text
本次入库后，服务器中疾病、实体、关系各增加多少；硬闸门是否仍为 0；哪些数据不能进入正式 CDSS。
```

通过口径：

1. 必须按本批次 `batch_id` 查询服务器真实数据。
2. 阻断项为 0 才能写“已入库并通过”。
3. 阻断项不为 0 时，状态只能写“已入库待收口”或 blocked，并继续修复后复测。

## 5. 旧候选池复用闸门

旧候选池只能进入以下两类结果：

| 结果 | 后续动作 |
|---|---|
| 重抽取线索 | 回到当前教材/PDF原文重新抽取 |
| 不建议复用 | 不进入后续归并流程 |

禁止输出“可直接入库”“可直接生成 delta”。若确需复用单条候选，必须附当前原文证据、来源页码/段落、实体槽位和审计结果。
