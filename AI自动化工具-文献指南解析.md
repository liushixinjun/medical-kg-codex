## 1. 核心原则

| 序号 | 原则 | 执行要求 |
|---:|---|---|
| 1 | 基础权威教材/专科专著搭骨架 | 先抽取疾病概述、病因、机制、临床表现、检查、检验、诊断、鉴别、风险分层、治疗、并发症、随访、预后和预防。 |
| 2 | 指南/共识/路径填决策血肉 | 补充适用人群、触发条件、禁忌、剂量、时机、推荐等级、证据等级、随访要求和可执行规则。 |
| 3 | CDSS 标准字典优先 | 诊断、症状、体征、检查项目、检查发现、检验项目、检验细项、检验标本、药品、手术、治疗项目、生命体征、医学术语和别名优先使用 CDSS 有效标准记录。 |
| 4 | 标准值直接使用 | 命中标准字典后，主名称、标准编码、UUID、基础人群限制直接取字典原值；不得再创建同义“标准化实体”。 |
| 5 | 未命中先注册 | 标准字典缺失时进入注册队列；未注册内容不得作为正式诊断、医嘱或 CDSS 推荐动作。 |
| 6 | 疾病三层固定 | 顶层学科 → 疾病大类 → 临床疾病；待分型疾病和具体分型均使用 `Disease`，通过 `has_clinical_subtype` 区分。 |
| 7 | 事实与推荐分开 | 疾病知识可以浏览；正式推荐必须由患者条件触发并连接标准动作、证据和主依据指南。 |
| 8 | 证据必须可追溯 | 定义、诊断、鉴别、阈值、治疗、禁忌和推荐必须定位到原文、页码/段落和来源。 |
| 9 | 同义词持续沉淀 | 标准全称作主名，中文简称、英文缩写、旧称放 `aliases` 并同步术语字典；不得重复建同义节点。 |
| 10 | 全量闸门代替抽检 | 每批次跑完整结构、主数据、证据和临床可用性审计，不能用少量抽检宣布合格。 |
| 11 | 写库受控且可回滚 | 写库前有备份/回滚清单，写库后做服务器全量复核；阻断项不为 0 不得标记完成。 |
| 12 | 错误形成复利 | 新问题同时写入步骤记录、踩坑日志和错误指纹索引，并转换为自动闸门或明确规则。 |

## 3. 统一执行流程

### G0：启动预检

1. 确认学科、疾病大类、目标病种、资料路径、输出路径和批次编号。
2. 锁定 Schema V2.0 和本技能 V3.0，不允许运行中临时切换版本。
3. 读取 `scope_taxonomy.csv` 和 `controlled_vocabulary.csv`；二者是启动硬门槛。
4. 读取标准字典表配置，只取有效标志为 1 的记录；检查 UUID、编码、名称和基础人群字段。
5. 对来源资料按“同一内容不同格式”去重，PDF、DOC、DOCX 同名同内容只计算一份。
6. 重大 Schema 迁移或全库治理前先备份 Neo4j；普通病种增量批次生成本批次回滚清单。

通过条件：范围、路径、版本、标准字典、来源清单和回滚策略全部明确。

### G1：资料解析、骨架和决策抽取

1. 教材按章节目录识别疾病边界，逐章抽取基础骨架，不跨章拼接临床规则。
2. 指南按页/段解析，保留原文、页码、章节、表格标题和来源信息。
3. 先生成疾病层级，再生成医学事实，再生成路径、规则、推荐和证据链。
4. OCR 低质量页面必须重新渲染/OCR 或人工核对；不能以缺失内容直接结束。
5. 一份指南只建一个 `Guideline`；同来源、同页码、同原文只建一个主 `Evidence`。

通过条件：来源清单、章节索引、节点候选、关系候选和证据索引完整。

### G2：标准字典归一与本地质量审计

1. 先匹配 CDSS 标准字典，再决定是否生成正式实体。
2. 生成标准字典快照、直接引用清单、未命中注册队列和别名增量。
3. 运行疾病层级、检查/检验归属、重复实体、诊断标准下钻、鉴别规则、治疗动作链、推荐证据链和来源裁决审计。
4. 阻断问题能根据原文和标准字典确定修复时，必须在本批次内修复并复测；不能修复时明确标记 `blocked`，不得伪造数据。

通过条件：所有入库硬闸门为 0；仅知识展示的非阻断项有明确状态和原因。

### G3：受控写入图谱数据库

只有同时满足以下条件才允许写库：

1. 用户已授权，或 `neo4j_action=import_after_audit`。
2. G2 全量审计通过。
3. 节点/关系增量包、标准字典引用、审计报告和回滚清单齐全。
4. 写库脚本先做“只演练不写库”验证，再执行正式模式。
5. 外部模型、前端、Trae 和人工 SQL/Cypher 不得绕过本地包直接修改正式 Neo4j。

### G4：服务器入库后复核与交接

1. 从服务器重新统计节点、关系、标签和关系类型，不引用本地预计数量。
2. 按批次编号核对实际写入数量、标准字典 UUID、疾病层级和证据链。
3. 运行服务器入库后复核；阻断项必须全部为 0。
4. 更新批次台账、步骤记录、踩坑日志、错误指纹和当前交接文件。
5. 提供可导入包、回滚清单、复核结果和下一批次建议，避免会话中断后无法继续。

## 4. CDSS 标准字典优先流程

### 4.1 必须处理的标准对象

每批次必须识别并处理以下 14 类，不得用“其他”省略：

1. 标准诊断；
2. 症状；
3. 体征；
4. 检查项目；
5. 检查发现/观察指标；
6. 检验项目；
7. 检验细项；
8. 检验标本；
9. 药品；
10. 操作/手术；
11. 标准手术；
12. 治疗项目；
13. 生命体征项目；
14. 医学术语和别名。

### 4.2 字典匹配顺序

| 顺序 | 匹配方式 | 处理结果 | 示例 |
|---:|---|---|---|
| 1 | 有效 UUID/标准编码精确匹配 | 直接使用标准记录。 | I21.900 |
| 2 | 标准名称精确匹配 | 直接使用标准名称、编码、UUID。 | 急性心肌梗死 |
| 3 | 已审核别名精确匹配 | 使用别名对应的标准实体，原词进入 `aliases`。 | AMI → 急性心肌梗死 |
| 4 | 规范化文本候选匹配 | 进入待审核匹配表，不直接正式使用。 | 经皮冠脉介入 → 经皮冠状动脉介入治疗 |
| 5 | 未命中 | 进入标准字典注册队列。 | 新检查发现术语 |

`standard_dictionary_mapping.csv` 只是审计和同步记录，不在 Neo4j 额外建立一层“映射实体”。命中后图谱临床实体直接使用 CDSS 标准值；只有疾病知识与编码、手术知识与编码确有不同语义时，分别使用 `Disease -> StandardDiagnosis` 和 `Procedure -> StandardProcedure`。

### 4.3 标准字典写入要求

1. 只允许使用有效标志为 1 的记录。
2. 主名称、标准编码、UUID、性别、年龄、妊娠、哺乳和有效期直接复制字典原值。
3. 不允许大模型推测字典字段枚举；实际编码与中文含义同时保留。
4. 医院自定义诊断和院内映射表不参与文献抽取；它们只用于上线医院的诊断/医嘱回填。
5. 凭据文件、密码和连接串不得写入 Git、日志、报告和图谱属性。

## 5. 疾病层级抽取规则

### 5.1 三层结构

```text
Specialty 顶层学科
  -> DiseaseCategory 疾病大类
    -> Disease 待分型/宽口径疾病
      -> Disease 具体临床分型
```

### 5.2 判定规则

| 对象 | 何时建立 | 诊断用途 | AMI 示例 |
|---|---|---|---|
| `DiseaseCategory` | 用于组织一组相关疾病。 | 不可诊断。 | 冠心病 |
| 宽口径 `Disease` | 临床可先作疑似/初步诊断，但尚未完成分型。 | 可作为待分型诊断。 | 急性心肌梗死 |
| 分型 `Disease` | 有明确诊断标准、标准诊断或临床分型意义。 | 可作为具体诊断。 | STEMI、NSTEMI |
| `DiseaseSubcategory` | 多个疾病需要稳定共享展示分组。 | 不可诊断。 | 急性冠脉综合征 |

硬规则：

- 不再把 STEMI、NSTEMI、肥厚型心肌病、扩张型心肌病建成 `DiseaseClassification`。
- 疾病阶段、并发症、术后状态不得误建成疾病分型。
- 宽口径疾病和分型疾病都要分别匹配有效 `StandardDiagnosis`；没有有效编码时 `is_emr_writable=false`。
- 前端可把同名目录和宽口径疾病折叠显示，但数据库中必须保持目录与诊断实体分离。
- 展示分组只允许使用 `DiseaseCategory -> has_display_group -> DiseaseSubcategory -> groups_disease -> Disease`；不得再生成反向归属关系。
- 核心诊断层级只允许使用 `Specialty -> has_disease_category -> DiseaseCategory -> has_disease -> Disease -> has_clinical_subtype -> Disease`，同一语义不得同时保留正向和反向关系。

## 6. 检查和检验抽取规则

### 6.1 检查

```text
Disease
  -> requires_exam_item -> ExamItem 检查项目
ExamItem
  -> exam_item_has_observation -> ExamObservation 检查发现
```

示例：急性心肌梗死 → 常规心电图 → ST 段抬高、病理性 Q 波、T 波倒置。

### 6.2 检验

```text
Disease
  -> requires_lab_item -> LabItem 检验项目
LabItem
  -> lab_item_has_subitem -> LabSubitem 检验细项
LabItem/LabSubitem
  -> lab_item_uses_specimen -> LabSpecimen 检验标本
```

示例：急性心肌梗死 → 心肌损伤标志物检测 → 心肌肌钙蛋白 I、心肌肌钙蛋白 T、肌酸激酶同工酶。

硬规则：

- `ExamItem` 不是检查结果；`ExamObservation` 不是可开立检查项目。
- `LabItem` 不是检验细项；`LabSubitem` 必须有所属项目。
- V1.x 的 `ExamIndicator` 必须根据父关系和原文拆为检查发现或检验细项，禁止机械统一迁移。

## 7. 基础人群过滤与临床规则

### 7.1 标准属性预过滤

适用于标准诊断、标准手术、药品、检查项目、检验项目和治疗项目：

1. 有效标志和生效/失效时间；
2. 性别限制；
3. 最小年龄、最大年龄和年龄单位；
4. 妊娠限制；
5. 哺乳限制。

### 7.2 规则引擎过滤

以下内容不得塞进简单属性，必须建立 `ClinicalRule` 或 `Contraindication`：

1. 肾功能和肝功能；
2. 过敏史；
3. 活动性出血和出血风险；
4. 当前用药和药物相互作用；
5. 生命体征和检验阈值；
6. 急性期、稳定期和疾病阶段；
7. 门诊、急诊、住院、重症监护场景；
8. 发病、检查、治疗时间窗；
9. 既往手术、卒中和重大病史；
10. 并发症和合并症。

## 8. 教材骨架抽取标准

教材章节常见槽位如下；章节没有某槽位时记录“不适用”或“来源未覆盖”，不得伪造：

1. 疾病概述/定义；
2. 病因；
3. 发病机制；
4. 病理生理；
5. 流行病学；
6. 临床表现：症状和体征分开；
7. 实验室检查：检验项目、检验细项和标本分开；
8. 辅助检查：检查项目和检查发现分开；
9. 诊断标准及其明细；
10. 鉴别诊断对象和鉴别规则；
11. 分型、分级和危险分层；
12. 治疗方案、具体动作、禁忌和人群；
13. 并发症；
14. 随访和预后；
15. 预防和患者教育。

教材骨架可以提供稳定基础知识，但不得把教材中的建议等级写成指南推荐等级。教材正文若直接引用指南等级，原文保留在 Evidence，同时推荐等级仍归属被引用的指南来源。

## 9. 指南 PDF 证据与正式推荐

### 9.1 Evidence 证据片段

每条关键证据至少保存：来源文档、页码、章节、原文、中文翻译/摘要、文档哈希和抽取时间。原文不得被摘要覆盖；同一来源、页码和原文必须复用同一 Evidence。

### 9.2 推荐动作链

```text
ClinicalPathway
  -> PathwayStage
    -> ClinicalRule
      -> RecommendationStatement
        -> recommends_action / blocks_action
          -> Medication / Procedure / ExamItem / LabItem / TreatmentItem / FollowUp
        -> derived_from -> Evidence
        -> based_on_guideline -> Guideline
```

### 9.3 多指南来源裁决

1. 大模型抽取候选，不直接决定主依据。
2. 按专病相关性、本土适用性、有效期、权威等级、推荐等级、证据等级和目标人群进行可复核评分。
3. 多指南一致时生成一条综合推荐，默认展示主依据，支持依据折叠展示。
4. 指南冲突时记录冲突条目和处理结果；冲突待裁决不得进入正式推荐区。
5. 新指南发布时只重抽受影响证据和推荐，不重建已冻结教材骨架。

## 10. V1.x 数据迁移规则

1. 迁移前必须完成数据库全量备份并校验节点、关系、索引和约束。
2. 先生成只读盘点和迁移映射，不直接写库。
3. `DiseaseClassification` 按临床语义分别迁移为 `Disease`、`DiseaseSubcategory` 或分类标签。
4. `Exam` 迁为 `ExamItem`；`LabTest` 迁为 `LabItem`。
5. `ExamIndicator` 根据所属项目拆为 `ExamObservation` 或 `LabSubitem`。
6. 旧诊断编码节点迁为 `StandardDiagnosis`；旧手术编码节点迁为 `StandardProcedure`。
7. `has_category` 改为 `has_disease_category`；`belongs_to_category` 改为疾病大类正向 `has_disease` 后删除。
8. `has_subcategory` 改为 `has_display_group`；`belongs_to_subcategory` 改为展示分组正向 `groups_disease` 后删除。
9. 关系迁移后先标记旧节点为已替换，确认无引用且回归测试通过后再进入物理删除候选。
10. 实体 `code`、关系 `source_code` 和 `target_code` 必须是单值字符串；历史合并编码写入 `merged_from_codes`，禁止把数组写入编码字段。
11. 全量迁移必须在一个数据库事务内完成，并在事务提交前运行迁移硬闸门；任一闸门失败必须自动回滚。
12. 旧层级关系盘点必须覆盖所有来源实体和所有重载用法，旧 `DiseaseClassification` 孤立节点也必须进入清理清单和回滚包。
13. 每次迁移只处理一个可回滚批次；服务器独立复核通过后再继续下一批。

## 11. 本地和服务器硬闸门

以下 18 项必须全量检查：

1. 非 `KGNode` 节点；
2. 非标准实体类型或关系类型；
3. 疾病大类被当作诊断；
4. 疾病无大类、分型无父疾病、分型环路和同名疾病重复；
5. V1.x 禁用实体和关系新增；
6. 标准诊断/标准手术缺 UUID、编码、名称或有效标志；
7. 标准主数据与 Oracle 有效记录不一致；
8. 症状与体征混用；
9. 检查项目与检查发现混用；
10. 检验项目与检验细项混用或细项无归属；
11. 技术编码、英文缩写或生成前缀成为医生可见主名；
12. 同类型同标准名/同 UUID 重复；
13. 诊断标准无明细；
14. 鉴别诊断无鉴别规则；
15. 治疗方案无场景、规则、具体动作或证据；
16. 正式推荐缺动作、主证据、指南或来源裁决；
17. 基础人群限制由模型猜测或与标准字典不一致；
18. 旧节点仍被正式查询引用、或前端把目录/阶段菜单当正式推荐。

任何阻断项不为 0 时，不得写“正式 CDSS 可用”。单个非阻断问题不得拖住其他可独立处理内容，但必须登记、隔离并在批次结项前明确去向。

## 13. 记录和交接闭环

每次任务完成后按“最新内容在最前”更新：

1. `批次登记台账.md`：范围、资料、实际使用文献、数量、状态和服务器结果；
2. `AI自动化工具-文献指南解析步骤记录.md`：用户问题、执行方案、结果和关键文件；
3. `_全局复利与踩坑日志.md`：根因、修复、自动防复发措施；
4. `项目管理中心_project_management/05_全局错误指纹索引_error_fingerprint.csv`：可机器识别的错误指纹；
5. `当前进度与后续计划_交接.md`：当前阶段、已完成、未完成、下一步和禁止事项。

已经写入本技能硬规则且具备自动检查的历史错误，不在主日志重复长篇描述；日志只保留新证据、影响批次和规则升级记录。

## 14. 禁止事项

- 不允许跳过标准字典直接创建正式诊断、药品、检查、检验、手术和治疗主数据。
- 不允许把医院自定义诊断或院内项目映射表当作文献抽取标准源。
- 不允许新增纯英文文件夹名；目录使用简短中文名，确有接口需要时可加短英文后缀。
- 不允许使用 `DiseaseClassification`、`Exam`、`LabTest`、`ExamIndicator` 生成 V3.0 新数据。
- 不允许把症状和体征、检查和检验、项目和结果混在一个实体类型。
- 不允许把英文缩写作为标准主名；缩写必须进入别名。
- 不允许候选、外部模型结果、Trae 修复结果绕过原文证据和审计直接写 Neo4j。
- 不允许用“抽检通过”“大致完成”“基本稳定”替代全量硬闸门结果。
- 不允许在日志、Git、报告或图谱属性中保存数据库明文密码。
- 不允许在医生端展示内部节点 ID、技术编码名、全量 Evidence 池或阶段可选动作冒充正式推荐。
